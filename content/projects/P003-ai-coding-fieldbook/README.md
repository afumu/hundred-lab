# AI 编程实战经验库

AI 编程实战经验库是百创计划记录的第三个正式开源项目。

这个项目当前还处于项目定义阶段，先不急着做成复杂产品，而是优先把真实使用 AI 编程工具过程中的经验、问题、复盘和方法沉淀成一个长期更新的公开经验库。

## 项目定位

这个项目的目标不是再做一个通用知识库，而是专门记录：

- 使用 AI 编程工具的真实工作流
- 在需求分析、计划、编码、Review、PR 各阶段遇到的问题
- 哪些能力应该交给 Agent，哪些能力应该由人来把控
- 哪些流程、文档和约束应该被提前沉淀
- 哪些地方会浪费 token、造成混乱或降低稳定性

它更像一个持续进化的 fieldbook：

- 记录一线经验
- 整理失败案例
- 提炼稳定方法
- 逐步升级为开源模板、技能、文章和产品

## 当前主线

当前最先记录的一条主线，是使用 Superpowers skill 过程中的真实体会。

当前观察到的基本流程是：

```text
需求分析
  -> 编写计划
  -> 编写代码
  -> review
  -> pr
```

这个项目会沿着这条链路，持续记录每个阶段到底存在哪些问题。

## 第一批问题

### 1. 需求分析

当前最明显的问题是：

- 有些需求已经足够明确，Agent 仍然会继续追问
- 在不需要继续澄清的情况下，依然停留在提问阶段
- 这会拖慢进入实现的节奏

这说明后续应该沉淀：

- 什么情况下需求已经足够明确，可以直接开始开发
- 什么情况下追问才是真正必要的
- 如何减少无效澄清

### 2. 编写计划

当前最明显的问题是：

- 文档管理混乱
- 默认只会生成 spec 文档
- 不会自动生成整个项目层面的文档结构
- 写完代码后，文档通常也不会同步更新
- 缺少统一索引，比如 `AGENTS.md`、`claude.md`、项目级文档入口

这说明后续应该沉淀：

- 项目级文档骨架
- 模块级文档拆分方式
- 文档索引规则
- 代码变更后的文档同步机制

### 3. 编写代码

当前最明显的问题是：

- 开启子代理整合后，容易在编码和 review 之间频繁切换
- 这种来回切换会消耗大量 token
- 对已有代码复用不够
- 明明项目里已经有现成实现，Agent 仍然喜欢自己再写一套

这说明后续应该沉淀：

- 什么情况下适合开子代理，什么情况下不适合
- 如何约束 Agent 优先复用现有代码
- 如何在编码阶段减少上下文切换

### 4. Review

当前这部分还需要继续补充更多一线观察。

后续重点可以关注：

- Review 到底是查逻辑问题，还是查风格问题
- 哪些 review 可以交给 Agent，哪些 review 必须人类亲自做
- review 输出应该如何结构化
- review 之后怎样把经验沉淀下来

## 为什么值得做

这个项目值得做，不是因为“AI 编程”这个词热门，而是因为它确实存在一套正在快速成形的新工作流，但这套工作流现在还很混乱。

真正缺的不是再多一个 prompt，而是：

- 一套可验证的经验库
- 一套真实可复用的流程
- 一套不断更新的错误与改进记录

如果这件事做好，后续可以自然衍生出很多资产：

- AI 编程工作流技能
- AI 编程项目模板
- 文档骨架生成器
- harness 工程方法论
- 系列文章和课程
