# 2026-04-19 Idea

## 主题：AI 编程 Harness、Skill 与项目规范

### 原始想法

- 今天大概用了四个账号，每个账号应该都用了两轮以上，也就是说切了八次。
- 失败之后会追问“what capability is missing”，这体现的是 harness 思维。它不是只优化当前任务，而是在为未来所有类似任务修系统。
- 以前人类需要看代码、看结果。现在是否还需要看？在什么情况下必须具体去看代码？架构层面是否仍然需要人类把控？
- 给任务时要同时给明确的测试指标、测试方式、接口位置。这部分应该形成接口文档，方便后续测试。
- 要先让 agent 列计划，然后我们先看计划。
- 对整体架构一定要有把握。实现原理、人类理解、技术实现路径都要清楚。
- 如果涉及测试、升级和报错，需要形成文档。
- 给一个大需求但方案没设计好，写出来就容易有问题。方案阶段必须让 agent 论证方案是否合理、哪里可能出问题，并把需求、流程和技术实现讲清楚。
- 技术方案的整体把握非常重要。要把实现思路、步骤和缺点讨论透。
- 测试很重要，尤其是先测试后端接口。
- 大方向和大的技术路线没问题，才可以继续放手执行。
- 还没有认真让 agent 操作浏览器，后续要测试一下 browser use 的效果。
- agent 能看到什么，基本决定了它能自治到什么程度：
  - 代码可见，只能做静态修改
  - UI 可见，可以做交互验证
  - 日志和指标可见，可以做性能和可靠性优化
  - trace 可见，可以做链路级诊断
- 整个项目应该生成 Wiki 文档。
- 应该让 agent 从一个小而稳定的入口开始，逐步知道去哪里找更深的信息，而不是一上来把所有信息砸给它。
- 给 Agent 一个目录让它自己找信息，往往比把所有信息直接塞给它更好。渐进式披露很关键。
- 和 LangGraph、LangChain 这类固定流程相比，Skill 这种渐进式披露很多时候更强。人为固定流程会让 agent 变重、变麻烦、传递成本变高。
- 最近几天在做一个 Agent 项目，之前用 Skills，这两天改成类似 LangChain 框架，发现效果反而更差。Skill 让模型自己决定什么时候调工具、什么时候跑流程，效果更好。
- 想思考能否给整个项目加一个 hook，让每次对话结束后自动把学习到的东西放进目录里。
- 关键问题是：什么时机让它去学习、去记录最合适？
- 现在会同时让它解决当前问题，又让它总结经验。总结经验可能污染后面的上下文，于是又需要回到前面的上下文继续回答问题。是否可以有一个机制，让它直接回到前面的上下文？
- 一些流程性的东西、执行命令相关的东西，可能还是需要我自己亲手执行一次。只有人真实走过一遍，才能发现流程问题，更有助于后续稳定发布。
- 真的可以开多个 watch，并行对话、并行执行任务，这可能才是完整态。
- 可以做一个 Harness 工程师培训系统，教人一步步成为 Harness 工程师。
- 整个项目的代码风格还是要先确定，否则写的时候容易出现风格偏差。
- 今天账号数量：3。

### 为什么值得记

- 这不是零散 prompt 技巧，而是一整套 AI 编程的工作方法雏形。
- 里面同时包含了任务设计、测试契约、架构边界、上下文组织、技能机制、学习机制和并行协作。
- 这些想法后续有机会长成产品、技能、项目、文章和视频，但当前还不必强行确定唯一方向。

### 可能方向

- `coding-harness` 技能
- AI 编程项目的 Wiki / Harness 骨架
- “Skill 为什么比固定框架更强”的文章或视频
- Harness 工程师培训系统
- 对话后自动学习和文档更新机制

### 暂不明确的点

- 这些想法里哪一条会优先升级为真正的产品
- “自动学习”的最佳触发时机
- “回到前面上下文”的机制应该如何设计
- 浏览器可见性在真实工作流里能带来多大提升

### 下一步观察

- 继续记录真实 AI 编程过程中的失败、返工、切号和成功案例
- 单独观察：哪些问题是方案设计不清导致的，哪些问题是可见性不足导致的
- 单独观察：Skill、固定流程框架、浏览器操作三者在任务质量上的差异

## 主题：9 个新 Idea 整理

### 整理后的方向

#### Idea 01：OpenHooks 实战经验库

- 核心想法：先把 OpenHooks 的真实实践、踩坑、范式和经验沉淀成经验库，不急着直接做成 Skill。
- 可能形态：经验库、案例库、后续再升级为 Skill 或开源产品。

#### Idea 02：百创计划自进化 Skill 系统

- 核心想法：把百创计划本身做成一个自进化的 Skill 系统，让它自己去 GitHub、开源社区、Hacker News、Twitter 等地方找需求，自己发散、自己判断产品方向、自己形成新产品或新 Skill。
- 可能形态：自进化 Skill、自动研究员、自动产品发现系统。

#### Idea 03：读书产品

- 核心想法：做一个围绕“读书、思考、记录、发散连接”的产品，把你的读书笔记、观点、片段、文章、关联思考都沉淀在一个地方。
- 可能形态：个人读书系统、知识图谱型读书产品、读书内容发布平台。

#### Idea 04：日常使用 AI 的开源项目

- 核心想法：把自己日常使用 AI 的想法、技巧、范式转换、操作方式系统化，做成一个开源项目。
- 可能形态：AI 使用手册、AI 工作流仓库、AI 范式实践项目。

#### Idea 05：自动切换 PPT 的工具

- 核心想法：在自己读稿或演讲的时候，系统自动生成节奏、自动翻页、自动切 PPT，不需要手动点击。
- 可能形态：演讲辅助工具、提词器 + 自动翻页产品、PPT 自动控制器。

#### Idea 06：微信读书浏览器插件

- 核心想法：当你在读某本书时，插件自动把当前内容和你已有的资料、笔记、知识点做关联，帮助发散思维。
- 可能形态：微信读书插件、阅读关联助手、知识连接插件。

#### Idea 07：多语言开源项目模板库

- 核心想法：做各种语言、各种项目类型的统一风格开源模板，例如 Java、Go、前端、前后端管理后台等，让开发者下载后可以快速起步。
- 可能形态：模板仓库、项目脚手架、工程初始化工具。

#### Idea 08：方法论 / 课程产品化 AI 平台

- 核心想法：把很多方法论、课程和经验做成 AI 平台和产品，而不是只停留在文档或课程本身。
- 可能形态：AI 课程平台、方法论产品平台、可交互学习系统。

#### Idea 09：图片内容读取与索引项目

- 核心想法：读取你所有图片里的内容，通过图像理解和意图识别，提取关键项目、关键语言、关键点，再统一存放起来。
- 可能形态：个人图片知识库、视觉内容索引系统、图像理解资产库。

#### Idea 10：B 站收藏读取与总结项目

- 核心想法：把 B 站收藏内容全部拉下来，支持查询、提问、总结和再利用。
- 可能形态：B 站收藏知识库、视频收藏整理工具、个人视频资产问答系统。

  - B 站收藏读取与总结项目
